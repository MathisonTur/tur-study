1. 商品规格表

源码的商品规格表：

CREATE TABLE `li_specification`  (
  `id` bigint NOT NULL COMMENT 'ID',
  `create_by` varchar(255) CHARACTER SET utf8 COLLATE utf8_general_ci NULL DEFAULT NULL COMMENT '创建者',
  `create_time` datetime(6) NULL DEFAULT NULL COMMENT '创建时间',
  `delete_flag` bit(1) NULL DEFAULT NULL COMMENT '删除标志 true/false 删除/未删除',
  `update_by` varchar(255) CHARACTER SET utf8 COLLATE utf8_general_ci NULL DEFAULT NULL COMMENT '更新者',
  `update_time` datetime(6) NULL DEFAULT NULL COMMENT '更新时间',
  `spec_name` varchar(255) CHARACTER SET utf8 COLLATE utf8_general_ci NULL DEFAULT NULL COMMENT '规格名称',
  `store_id` varchar(255) CHARACTER SET utf8 COLLATE utf8_general_ci NULL DEFAULT NULL COMMENT '所属店铺',
  `spec_value` text CHARACTER SET utf8 COLLATE utf8_general_ci NULL,
  PRIMARY KEY (`id`) USING BTREE
) ENGINE = InnoDB CHARACTER SET = utf8 COLLATE = utf8_general_ci ROW_FORMAT = DYNAMIC;

重构后的商品规格表：

CREATE TABLE `goods_spec`  (
	`id` bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '主键id',
	`create_by` varchar(255) not null comment '创建者',
	`create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
	`update_by` varchar(255) not null comment '更新者',
	`update_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
	`delete_flag` boolean not null default 0 comment '是否删除',
  	`name` varchar(50) unique not null COMMENT '规格名称',
  	PRIMARY KEY (`id`) USING BTREE
) comment '商品规格表';
CREATE TABLE `goods_spec_value`  (
	`id` bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '主键id',
	`create_by` varchar(255) not null comment '创建者',
	`create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
	`update_by` varchar(255) not null comment '更新者',
	`update_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
	`delete_flag` boolean not null default 0 comment '是否删除',
  	`spec_id` bigint unsigned not null COMMENT '规格id',
  	`store_id` bigint unsigned not null default 0 COMMENT '所属店铺 id 0:平台 不为0：店铺id',
  	`spec_value` varchar(100) not null comment '规格值'
  	PRIMARY KEY (`id`) USING BTREE,
 	index `idx_spec_id`(`spec_id`) USING BTREE,
    index `idx_store_id`(`store_id`),
    unique `uk_spec_value`(`pec_id`, `spec_value`)
) comment '商品规格表';

说明：

由于规格值使用text类型字段存储的话，不便于查询操作，为此将值和名称分成两张表，并做如上设计

重点说明一下 value 表中为什么要由 store_id

  • 为什么要添加所属店铺 id？

    每个店铺的商品规格不一定是相同，规格名大部分一样，但规格值差别很大

    如手机机身颜色，在相同的规格名下，不同的手机定义的机身机身颜色值相差很大。

  • 在平台运营端有规格列表管理

    可以理解为平台预定义了常见的规格以及规格值，便于商家选用，减少不同商家定义相同的一个

    如常见的鞋码、衣服尺寸等等

  • 如何区分平台定义的规格与商家自定义的规格？

    用 store_id 区分：

    • 为 0 时，是平台定义的
    • 不为 0 时为商家定义的

问题：如果存在两个商家恰好自定义相同的规格和值，需要考虑数据重复吗？此处为了避免业务复杂，不管，如果要解决的话就是商家添加规格值时，检测时是否存在相同规格名下相同的规格值，若是由相同的，将值的store_id 更改成 0，直接提取成平台自定义，共享给商家。但这样又会存在别的商家可以看到别的商家定义的规格值，虽然无伤大雅（毕竟规格值没什么私密的），但总感觉不好

重构后：

CREATE TABLE `goods_category`  (
	`id` bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '主键id',
	`create_by` varchar(255) not null comment '创建者',
	`create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
	`update_by` varchar(255) not null comment '更新者',
	`update_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
	`delete_flag` boolean not null default 0 comment '是否删除',
	`commission_rate` decimal(10, 2) not null COMMENT '佣金比例',
  	`image` varchar(255) not null COMMENT '分类图标',
  	`level` tinyint unsigned not NULL COMMENT '层级',
  	`name` varchar(255) not null COMMENT '分类名称',
 	`parent_id` bigint unsigned not null COMMENT '父ID 为0则代表是一级分类',
  	`sort` tinyint unsigned not null DEFAULT 0 COMMENT '排序值',
  	`is_enable` boolean not null default 1 COMMENT '是否启用',
  	PRIMARY KEY (`id`) USING BTREE,
    key `idx_parent_id` (`parent_id`) using btree,
    index `idx_is_enable`(`is_enable`) using btree
) comment '商品分类表';

对于源码做的更改如下：

  • 修改 level 的类型为 tinyint，而不用 int，这是因为层级只有三级，即数值很小，不需要使用 int 类型，tinyint 就足够了

  • 为了满足 parent_id 不为空，修改 parent_id 的用法为一级分类的父 id 是0

  • 排序值类型可以选择为 tinyint，从演示站点的数据来看，作者将禁用的分类的排序值设置成了 100，而正常启用的分类排序值在 1-10 之间。又鉴于一个商城的一级分类不会说会有超过 127，故而用 tinyint 更符合

  • 源码中的 support_channel 字段，虽然注释说明是是否支持通道，但看不明白，通过对比演示站点，推测这是用来说明是否启用的字段，为此，更改字段名为 is_enable， 更改类型为 tinyint

  • 增加 parent_id 的索引，因为需要通过 parent_id 查询所对应的父类id，方便整合分类树

注：此处的参数是分类下所拥有的参数，与商品参数有所不同

相比源码做的更改：

  • 不知道参数类型作何用的，源码的实体类中也没有，删除
  • 分类id字段貌似无用，因为会有一个分类与参数组关联表，而参数表中已经有了参数分组id，删除
  • 在 mysql 5.0 以后，varchar(n) 中的 n 代表字符，即可存储 n 个中文或字母或数字，故而此处更改参数名称长度为5，限制参数名称，参数名称不能太长。此处不讨论会有英文名称的参数名
  • 增加分组id索引，需要用参数分组 id 来查找参数

说明：为什么源码中会有原图路径、小图路径、缩略图路径？

13. 商品sku表

sku 解释：

sku指库存量单位，就是库存进出计量的基本单元，可以是以件，盒，托盘等为单位。sku是对于大型连锁超市配送中心物流管理的一个必要的方法，是客户拿到商品放到仓库后给商品编号、归类的一种方法。通俗来讲就是一个产品最小的出库单位，比如说一款手机产品有红黑白三个颜色，那么一台红色手机就是一个sku。那么多销售属性的产品，再往下分，比如说一件T恤除了有颜色还有尺码，那么一件s码绿色体恤就是单个sku。

设计想法：

由于这张表与商品详情表实在是太像了，不想再重复的添加字段，于是打算将商品模板表做成一个关联表，用来声明那个商品是模板，依靠 goods_id 去商品详情表获取相关信息，同时因为商家管理端中有关于商品名称和商品编号的查询条件，故而也添加这两个字段。当然这样业务会复杂点，如果想业务简单点的，直接照源码会更简单。

也有另一种想法，就是不要这张表，直接在商品详情表中添加一个声明是否是模板的字段，这种在数据库设计方面更简单，但既然商家管理端中都已经有了商品模板这一功能，还是给它建个表，以示尊重 (’∇’)シ┳━┳
