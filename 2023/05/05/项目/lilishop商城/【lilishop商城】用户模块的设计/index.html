相比于源码的主要修改说明：

  • 为 avatar 设置默认值

  • 删除 description 字段，貌似无用，且一个运营管理端用户，需要为自己附上一个简介，这不是社交用户

  • 删除 role_ids 字段，观察到还有一个用户角色表，这里不需要再次声明用户角色

  • 更改 sex 字段名为 gender，这个就是纯粹不想用 sex，而且 gender 语义更贴切。可不用修改

  • 修改字段 is_super 名为 is_super_admin ，避免因为去掉前缀后命名无法使用的缘故，需要再次去修改命名，而且is_super_admin 语义更贴切，去掉前缀转换时也无需修改，建议采用

    注：super 在 Java 中是关键字

  • 为字段 email、mobile、username 分别添加 unique，因为这三个字段的值，不同用户肯定不同，且不同用户不能有一样的值

合并admin_user和clerk表做的更改：

  • 为了使两张表的逻辑更加相似，删除 member_id 字段，商家端的店员不需要再是商城注册的会员，可以由商家自定义，添加逻辑同平台运营端的
  • 为组合字段 store_id、department_id 添加联合索引

说明：

为什么会有 is_default_role 字段？

在添加新运营端用户时若是没有指定用户角色，则需要从角色表取出默认的角色关联新添加的用户，但在源码中并没有看到有用到这个。٩(º﹃º٩)

而且若是操作不当，出现了两个默认的，虽说可以在查询时限制查询数量为1，但获取到角色不一定是想要，这就需要去维护角色表中只能有且只有一个默认的。同时若是操作不当，使得表中没有默认的角色，新添加的用户登录时的权限就会出事故。因此，字段 is_default_role 有必要，而且需要建立索引

为什么为 name 和store_id字段设置为唯一索引？

首先说明下源码中的问题：在新增/修改角色时没有校验角色名是否已存在，以及角色名中的空格影响，这就造成同名角色共存，不仅会造成数据重复，还会导致为用户分配角色时出现选择错误。

故而需要对 name 字段设置为 unique。同时又为了兼顾平台运营端和商家端，引入了 store_id 字段，由于两个不同端可以存在相同名称的角色，因此需要将 name 和 store_id 组合创建唯一索引

PS：源码中这个问题出现很多处，可以说是开发者没考虑到这一点，都没处理

    index `idx_role_id`(`role_id`),
    index `idx_store_id`(`store_id`)
) comment '管理端角色菜单关联表';

说明：为什么角色菜单关联表中要由 store_id 字段？

商家端不允许修改菜单，但可以为商家端的角色修改权限（即修改菜单关联），同时在菜单表中只区分平台与商家端，所有商家的菜单都一样，但角色权限不同，需要在关联表添加店铺 id ，来区分不同店铺
