【Lilishop商城】做项目前的思考

lilishop商城
项目
发布日期:   2023-05-01
更新日期:   2023-06-01
文章字数:   2k
阅读时长:   6 分
阅读次数:  

lilishop商城源码地址：https://gitee.com/beijing_hongye_huicheng/lilishop

为什么选择这个项目？

  • 这个项目有详细的开发文档、使用文档、架构文档。这意味着我们可以很好的理解源码，进而写出自己的项目

  • 全部源码开源，前后端分离的项目。前后端分离已经是大趋势。

    PS：如今看到不是前后端分离的项目，都不想去做[捂脸]

  • 项目涉及的业务很丰富。其中促销模块的秒杀、拼团、砍价等等这些功能非常适合来练手

  • 项目是 b2b2c 商城项目，比起一般的单商户商城复杂度要高上很多，更加适合来熟悉商城系统

个人想法：

项目是单体架构的，而不是微服务架构的，为了熟悉微服务架构，尝试将其重构，可以更好的熟悉微服务架构

学习步骤

提示：

在做任何项目之前都需要先好好想想该怎么去做项目，步骤是什么，要达到什么程度，如果是团队项目，需要思考充当什么角色，在了解业务需求后，如何进行任务划分。

1.功能尝试（如果有源项目参考的话）

（如果源码有演示地址）体验源码呈现的效果是怎么样的，或者自己拉取项目到本地，搭建环境，运行源码

提示：

自己拉取源码到本地进行运行时，可能会遇到不少错误，如果没时间的话，还是直接看演示地址的。当然，自己搭建环境运行源码更好

2. 需求分析

业务需求分析，可以做一个需求文档，说明项目需要哪些功能，每一项功能的需求及业务流程是什么样的。不过在本商城项目的文档已经有了使用文档（即需求文档），可以简单的了解业务需求

3. 项目结构与数据库表设计

对于本项目而言，在已有的源码的基础上，来确定项目的结构和数据库表

项目结构的划分：

  • 项目采用什么架构
  • 划分项目模块
  • 各个模块要实现的什么功能
  • 模块之间怎么调用

重点：根据业务需求设计数据库表

数据库表的设计：

  • 确定表的设计规范，如参考阿里的开发规范
  • 是否要采取模块数据库一对一
  • 字段类型定义
  • 索引定义
  • 表关联

个人想法：

不建议一开始便将项目全部的数据库表设计出来，而是根据模块或功能的划分，设计一个模块的数据库表，然后完成相关的业务，再设计其他的。如编写商品模块时，先只设计好商品模块相关的数据库表，后编写业务。要知道项目的数据库表是很多很多的

4. 确定接口文档

确定接口文档，请求方式、入参、出参、关联的数据结构和逻辑等

此时需要前后端一起确定接口内容，并且此时可能会涉及到数据结构调整

5. 编码

实现具体的业务功能

6. 简单接口测试

测试所实现的功能是否有bug，一旦发现立即修复。测试可以和编码同时进行。

技术选型

源项目的开发文档里已经列出了相关的技术选型，见：开发文档-项目介绍-技术选型。但还是自己想想需要什么技术，因为时间的关系，有很多技术都迭代更新了，我们应该拥抱新技术。再查找相关技术之间版本的关系，选定好版本，避免版本冲突（说多了都是泪），罗列出来：

技术 说明 版本
SpringBoot 基础框架 3.0.2
SpringCloud 微服务框架 2022.0.0
SpringCloudAlibaba alibaba cloud 组件 2022.0.0.0-RC2
MySQL 关系型数据库
sharding 数据库分库分表
Mybatis-Plus 持久框架
Spring Security 安全框架

Spring Cloud Loadblancer 负载均衡
Seata 分布式事务

ElasticSearch 数据搜索
XXL-job 任务调度

以上自己罗列的技术与原项目的有所差别，一切都要根据自己会使用的技术来定，而不是直接照搬原项目，虽然直接按照原项目的技术来更简单

SpringBoot、SpringCloud、SpringCloudAlibaba 这三个版本尤其需要配对：见：SpringCloudAlibaba 版本说明

提示：

明确的技术选型可以方便团队合作，谁也不想拉取的代码到了本地，运行起来一堆bug

数据库设计规范

以下内容参考阿里开发规范。

请一定务必必须按照规范，这里以阿里规范来设计。以阿里规范去看源码的sql，真的很揪心

源码数据库表存在的问题

  • 所设定的字符集如今不推荐使用，建议删除设置字符集语句，即不设置字符集，使用数据库默认
  • 存储浮点数使用的double类型，不推荐使用，改为使用 decimal 类型
  • 表示是否的字段使用 bit 类型，改为 tinyint 类型
  • 源码大量存在允许字段为 null 的设置，不建议。应该设置为 not null
  • 源码存在不少未在实体类中使用的字段
  • 源码中的表很多莫得注释

PS：不晓得原开发者为什么要将表中其他表的 id 的类型设置成字符串类型，bigint 类型不是更加方便使用吗？

表必备的几个字段

`id` bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '主键id',
`create_by` varchar(255) not null comment '创建者',
`create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
`update_by` varchar(255) not null comment '更新者',
`update_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
`is_delete` tinyint unsigned not null default '0' comment '是否删除'

必备的字段在每个表中都存在，因此，可以定义一个基类，包括以上字段，让其他实体类继承基类，减少重复代码

注：在 mysql 8.0 以后整数类型不用指定长度，已经被废弃
