平台运营端

1. 计量单位

看似是简单的 CRUD 操作。但仔细思考下有以下几个问题：

PS：由于计量单位的信息很少，VO 类与 entity 类几乎一样，源码中直接使用了 entity 类返回给前端，按照规范来说这是很不提倡的，但若是增加一个 VO 类，内容与 entity 类区别不大，纠结 ing

看一段 gpt 的回答：

如果您的 VO 类的属性与 Entity 类几乎一样，那么在某些情况下，确实没有必要新建 VO 类。但是，根据软件设计的单一职责原则，VO 类和 Entity 类的职责是不同的，它们的设计目的也不同。通常情况下，VO 类用于展示层，用于与前端交互，而 Entity 类用于持久层，用于与数据库交互。因此，即使属性相同，也建议将 VO 类和 Entity 类分开使用，以保证代码的可读性和可维护性。
如果您的 Entity 类的属性可以直接返回给前端，那么可以直接使用 Entity 类返回给前端。但是，如果您需要对 Entity 类的属性进行一些特殊处理，例如格式化日期、转换枚举值等，那么建议使用 VO 类，将 Entity 类的属性转换为适合前端展示的格式。这样可以保证前后端分离，降低耦合度，提高代码的可维护性。

1.1 查询

在分页查询单位数据时，为了方便前台人员的操作（删除操作），返回的数据中增加属性，使用字符串表示是否存在关联商品。（也可以不用，因为删除失败有提示的）

若是这单位是一定要删除的，为了方便操作人员快速的找到关联的商品，可以商品列表里添加计量单位的搜索选项，可以展示对应的计量单位的商品数组，方便处理

注意点：

  • 名称重复问题
  • 空值问题
  • 由于逻辑删除字段的存在，为了保证名称唯一，当存在相同名称的待删除（即 delete_flag=1）数据时，可以修改该数据的逻辑删除字段为未删除(即 delete_flag = 0)，而不需要添加新数据

    • 若为 0，则抛出名称已存在异常

1.3 修改

注意点：

  • 名称不允许重复
  • 空值问题
  • 由于逻辑删除字段的存在，当修改后的名称与待删除的数据同名时，为了保证名称的唯一，此时需要删除待删除的同名的数据。这里采取删除待删除的同名数据，是为了保证商品关联的单位 id 没有发生变化，避免麻烦。这点跟添加是不一样的，注意

源码：

  • 源码中修改计量单位时，未修改关联了该计量单位的商品计量单位名称

  • 测试发现，当修改一个单位名称与其他已存在的单位名称一样时，可以修改成功，并没有抛出错误信息，出现了两个名称一样的计量单位。从源码中可以看到在编辑单位名称时并没有检测重复的逻辑：

想法：

  • 更改商品计量单位的字段为 unit_id，而不是 unit 的名称。这是考虑到若是使用的字段是计量单位的名称，那么在计量单位名称被修改时，还需要在商品详情表中找到计量单位为被修改的数据，修改计量单位名称，这显然很麻烦，效率也低，故而更改字段为 unit_id，使得计量单位的修改不用去修改详情表的数据，更加便利。
  • 名称不允许重复、空值检测
  • 名称检测重复
    • 若时遇到同名的且为待删除的数据，则删除。
    • 若是存在未删除的同名的数据，则抛出异常

1.4 删除

删除：

删除一个计量单位后，关联了这个计量单位的商品中的计量单位怎么办？

源码中就直接删除了，没做任何处理。(▼ヘ▼#)。虽然源码中商品详情表是直接使用了计量单位的名称，而不是计量单位的 id。这好吗？这很不好。而且表中既然已经有了逻辑删除字段，就应该设置逻辑删除值来达到删除目的，而不是直接删除，真正删除时借用定时任务来完成

想法：

在删除计量单位时，搜索一下商品详情表，看是否有商品关联了该单位，若是有，则警告不允许删除，必须需要等到没有商品关联该计量单位时才可删除，因此在商品详情表中需要对 unit_id 建立索引，提高搜索效率

2. 规格列表

2.1 查询

在分类列表中添加一个规格的搜索框，搜索关联该规格的分类，方便操作

PS：对于查询返回的数据到底该不该返回创建时间、创建者、更新时间、更新者？

GPT：这个问题需要根据具体业务需求来确定。
如果这些信息对业务有较大的作用，可以将它们包含在查询返回的数据中，方便业务进行相关操作和分析。例如，如果需要对数据进行审计或者监控，这些信息就非常重要。
但是如果这些信息对业务影响不大，或者不需要进行相关操作和分析，可以不将它们包含在查询返回的数据中，减少数据传输和存储的开销。
需要注意的是，如果这些信息包含敏感信息，例如创建者或更新者的账号信息，需要进行脱敏处理，以保护用户的隐私。
总之，是否需要将创建时间、创建者、更新时间、更新者包含在查询返回的数据中，需要根据具体业务需求来确定，避免不必要的数据传输和存储开销。

2.2 添加

注意点：

  • 规格名不允许重复

  • 名称、规格值不能为空（NotBlank）

  • 规格值重复的问题

  • 当存在同名的待删除的规格数据时，进行修改逻辑删除值重用

对于以上，源码的前端对规格名重复进行了检测，但后端还是需要再次检测。而空值问题没有解决。而对于规格值重复的问题的处理是在再次输入重复的值时，再按下空格后会清除输入的重复值的所有选项，这个体验不太好，可以改成提示有重复项并清除该项，且后端针对规格值项进行一次去重，再保存

想法：

  • 规格名称字段添加 unique

  • 名称进行空值、重复判断

  • 规格值进行去重

  • 规格值项不进行与其他规格的规格值项的重复检测。怕麻烦。但需要保持同一规格下，规格值唯一

  • 名称检测重复。

    • 不存在同名的直接添加

    • 检测同名的数据的逻辑删除值是否为 1

      • 若为 1，则可以直接修改 delete_flag=0，而不用直接添加数据
      • 若为 0，则抛出名称已存在异常

  • 规格值检测待删除的数据是否存在重复。若存在同名的则更新该数据为未删除，并且更新 spec_id 为插入的规格 id，以及 store_id 为插入 spec 时指定的 store_id

2.3 修改

注意点：

  • 如何判断规格名是否修改了

  • 修改后的名称重复检测

  • 空值问题

  • 修改的规格值项不允许重复

  • 修改后的规格值如何存储到数据库

  • 同名的待删除数据的重用

对于以上，源码中没有针对修改后的名称进行重复检测，空值问题也没处理。而对于重复值项的处理是输入重复值后无法添加进入值组。至于保存修改后的规格值，由于采用的表结构不一样，源码由于值为一个 text 类型字段，直接进行保存或添加了，而我将值分表，不能照着源码处理

想法：

  • 同添加操作

  • 规格名是否修改判断：根据规格 id，查询数据库中的 name，与修改数据中的 name 进行对比，不一样则修改，在修改时需要进行规格名是否重复的检测

  • 值修改：根据规格 id，从值表中查询所有值项，再与修改的值项进行对比，选出需修改的值项，再进行删除标记元数据中无用的值和添加修改的数据。

    PS：原本也想过先根据规格 id，删除对应的所有规格值，再进行添加新的数据。但这样容易造成 id 不足，特别是在数据量很大的情况下。

代码实现流程：

  1. 检测规格名是否修改。若修改了执行以下逻辑：

    1. 检测规格名是否重复
    2. 根据 id 更新规格名

  2. 更新规格值

    1. 规格值数组去重

    2. 根据规格 id 从数据库中获取旧的规格值数组（在 delete_flag = 0 的前提下查询）

    3. 批量删除标记无用的数据（即设置 delete_flag 的值为 1）

      1. 对比新旧两个数组，找出旧中包含而新中不包含的项，添加到一个数组，即为待删除的数组

      2. 当待删除标记的数组不为空时，执行批量删除操作

        注：批量删除使用的条件是 where id in (...)，括号内为待传入的id数组，若传入的数组为空，SQL语句会抛出异常的，需要在执行该语句前进行为空判断

    4. 批量插入新的规格值项

      1. 对比新旧两个数组，找出新中包含而旧中不包含的项，添加到一个数组
      2. 当待添加的数组不为空时，执行批量插入操作
        1. 检测是否存在同值的待删除数据
        2. 若存在同值的数据，则更新该数据为未删除，并更新 spec_id 为本次修改的规格的 id，若是 store_id 不同也更改相同

2.4 删除

注意点：

  • 删除标记时需要检测是否存在与之关联的分类
  • 批量删除与单个删除逻辑一样，可以合并为批量删除的逻辑
  • 批量删除时需要注意待删除的 id 数组不能为空
  • 删除标记设置而不是真正的删除

想法：

  • 删除标记时需要检测是否存在与之关联的分类时，若存在关联的分类，抛出异常给前端。由于在分类页面添加了可依照规格来搜索分类，这里就可以不用抛出包含分类名的异常信息了
  • 批量删除

3. 商品品牌

3.1 查询

需要根据品牌名、状态建立查询条件

3.2 添加

注意点：

  • name、logo 唯一，需要检测是否重复
  • name、logo不允许为空字符串
  • logo 格式的问题。一般图片的上传是从本地上传，可以不用管，依靠文件上传接口返回存储的图片的地址，但有时也需要直接输入图片链接，这时便要判断图片的格式是否符合

图片的格式的正则表达式如下：

^(http|https):\/\/[^\s]+\.(jpg|jpeg|png|gif)$

具体解释如下：

  • ^ 表示匹配字符串的开头。
  • (http|https)表示匹配 http或 https协议。
  • :\/\/ 表示匹配冒号和双斜杠。
  • [^\s]+ 表示匹配除空格以外的任意字符， + 表示匹配一个或多个。
  • \. 表示匹配一个点。
  • (jpg|jpeg|png|gif) 表示匹配 .jpg 、 .jpeg 、 .png 或 .gif 文件后缀名。
  • $ 表示匹配字符串的结尾。

难点1：本地上传的图片如何检测是否重复？

通过计算图片的哈希值来进行判断，具体步骤如下：

  1. goods_brand 表增加 logo_hash_value 字段，存储图片的哈希值，方便比较
  2. 获取上传的图片的字节数组，并计算哈希值
  3. 查询品牌表中是否已存在该哈希值，已存在则抛出异常说明图片已存在，不存在则可添加

难点2：当出现 name、logo同时出现匹配时（指的是同时匹配到两个数据，一个name同，一个 logo同），如何添加数据？

由于会出现两个数据，需要用数组来接收。而针对这个数组，就有以下几种情况：

  • 数组为空：直接添加数据
  • 数组长度为1：判断 delete_flag，待删除则更新数据，否则抛出异常
  • 数组长度为2：
    • 查询到的是同一个数据：判断 delete_flag，待删除则更新数据，否则抛出异常
    • 两个不同数据：
      • 均为待删除：删除一个，更新一个
      • 其余情况（均为未删除、一待一未）：抛出异常
  • 其余情况：抛出已存在异常

而使用分开查询后，得到的数据的情况有很多种：

  • 均为 null：直接添加数据

  • 一为null，一不为null：

    • ByName 的不为 null，ByLogo 的为 null：依据 ByName 的数据的 deleteFlag 值判断
    • ByName 的为 null，ByLogo 的不为 null：依据 ByLogo 的数据的 deleteFlag 值判断

  • 均不为 null

    • 两者相同：即为同一个数据。若是待删除的则直接更新删除标记。未删除的则直接抛出异常

    • 均待删除：删除一个，更新一个

    • 一个待删除一个未删除，均为未删除：抛出已存在异常

逻辑很复杂，不推荐使用

3.3 更新

注意点：

  • 名称、logo 查重
  • 名称、logo 不允许空值
  • logo 格式

想法：

  • 不存在同名或同logo的数据：直接更新数据

  • 存在同名或同logo的未删除的数据：直接抛出异常

  • 存在同名或同logo的待删除数据：删除该数据，执行更新待更新数据

对于以上逻辑的实现：使用数组来接收查询到的重复数据：

  • 数组为空：直接更新

  • 数组长度为1：若为待删除的数据则删除并执行更新，否则抛出异常

  • 数组长度为2：

    • 全为待删除：全删除，再更新

      注：有可能是同一个数据，若是执行两次根据主键id删除，就可能会抛出不必要的错误，需要在删除前进行查询，检测该数据是否存在

    • 存在未删除：抛出异常
